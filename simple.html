AWS Modernisation — Project Weekly Plan

PT Customer · Xtremax Engagement · 13-Week Delivery · 3 BAST Milestones

Month 1 · Foundation Month 2 · Build Month 3 · Deliver

Month 1 — Foundation

Week 1
Project Kick-off & Environment Discovery
Objective
Formally kick off the engagement. Understand the existing AWS environment, network topology, and application landscape. Provision Linux instances in Non-Production and begin exploring container build images for sample apps.
Key Activities
  • Kick-off meeting
  • Verify access to all AWS accounts
  • Map network topology
  • Finalise 6 application candidates
  • Provision Linux EC2 instances in Non-Production
  • Explore the build images for 2 sample apps
Pre-requisites
  • PO issued — engagement clock starts Day 1
  • AWS console credentials for all accounts provided before kick-off
  • 6 app owners introduced and available for interviews
  • Shadow engineers assigned and available from Day 1
  • Non-Production EC2 quotas and Linux AMI access pre-confirmed
Week 2
Network Setup, EKS Preparation & Image Testing
Objective
Continue building and validating container images. Set up the production network components and EKS prerequisites in Non-Production. Test images in the EKS environment by end of week.
Key Activities
  • Continue to explore the build images
  • Configure Security Groups for ALB, EKS, RDS tiers
  • Create EKS IAM service role and node instance profile in Non-Production
  • Start to setup the EKS environment in Non-Production
  • Test the images in EKS environment
Pre-requisites
  • Linux EC2 instances provisioned in Non-Production (from W1)
  • Initial build images for 2 sample apps identified (from W1)
  • VPC and subnet design confirmed with Customer IT lead
  • EKS IAM permission boundaries agreed
Week 3
DevOps Pipeline, IaC Foundation & Security Audit
Objective
Establish the DevOps pipeline foundation using GitHub and AWS native services. Set up Terraform remote state and module structure. Conduct DevSecOps workshop and begin security audit of the AWS organisation.
Key Activities
  • Start to setup DevOps pipeline using GitHub repo and AWS native services
  • Create ECR repositories
  • Set up Terraform remote state
  • Start to define Terraform module structure
  • Provide a workshop related to DevSecOps
  • Audit SCPs, OU structure, IAM users/roles, and MFA enforcement
Pre-requisites
  • EKS environment running in Non-Production (from W2)
  • Images tested successfully in EKS (from W2)
  • GitHub organisation access granted by Customer
  • DevSecOps workshop audience and schedule confirmed
Week 4
Landing Zone Enforcement, EKS Production Deployment & BAST 1
🏁 BAST 1 · 30%
Objective
Apply all AWS security guardrails from the W3 audit. Deploy the production EKS cluster via Terraform. Formally close Month 1 with BAST 1 documentation and Customer sign-off.
Key Activities
  • Enforce missing SCPs: prevent root usage, enforce MFA, block S3 public access
  • Enable AWS Config and org-wide CloudTrail
  • Enable GuardDuty
  • Configure VPC Flow Logs and Security Hub
  • Test to deploy EKS cluster via Terraform
  • Write and apply Terraform vpc module over existing VPC resources
  • Provide BAST 1 documentation
Pre-requisites
  • SCP audit findings from W3 reviewed and prioritised
  • Terraform remote state and module structure ready (from W3)
  • EKS IAM roles confirmed in Non-Production (from W2)
  • Customer stakeholder calendar blocked for BAST 1 presentation
🏁

BAST 1 — 30% Payment Milestone
Landing Zone Governance Report · OS Migration Documentation (6 apps) · App owner sign-off evidence · Security Hub FSBP compliance report

Month 2 — Build  ·  Weeks 5–9

Week 5
Production EKS Infrastructure Setup via Terraform
Objective
Deploy and validate the full production EKS cluster infrastructure via Terraform. Configure pod-level IAM access, namespaces, and Secrets Manager. Write the Terraform rds module ready for Week 8.
Key Activities
  • Write and apply Terraform eks-cluster module to production account
  • Install required EKS add-ons and validate all nodes are in Ready state
  • Create EKS namespaces for DEV, STG, and PRD environments
  • Configure IRSA — pod-level IAM access for apps via OIDC
  • Set up Secrets Manager — store app secrets and database credentials
  • Write Terraform rds module (database cluster, subnet group, parameter group)
Pre-requisites
  • BAST 1 signed — Month 2 work can begin
  • Terraform vpc module applied and state clean (from W4)
  • ECR repositories and GitHub repos ready (from W3)
  • EKS IAM roles confirmed in non-production (from W2)
Week 6
DevSecOps CI/CD Pipeline & Pilot Apps End-to-End
Objective
Build the full DevSecOps pipeline: GitHub → CodePipeline → CodeBuild (unit test + SonarQube + security scan) → ECR → CodeDeploy to EKS DEV → quality gate → manual approval → STG/PRD. Validate the full flow end-to-end using the 2 pilot apps.
Key Activities
  • Set up CodePipeline: GitHub source → CodeBuild → ECR → CodeDeploy to EKS DEV
  • Configure CodeBuild stage: unit test → SonarQube quality gate → container security scan → push to ECR
  • Configure automation test and quality gate stage after DEV deployment
  • Add manual pipeline approval gate (Team Lead / SPV) before STG/PRD promotion
  • Configure CodeDeploy to EKS STG/PRD after approval
  • Review and validate Dockerfiles and Helm charts for 2 pilot apps
  • Run 2 pilot apps through the full pipeline end-to-end — DEV to STG/PRD
Pre-requisites
  • EKS cluster with all namespaces running in production (from W5)
  • SonarQube server deployed and accessible from CodeBuild
  • CodePipeline and CodeBuild IAM roles created
  • Developer team has written Dockerfiles and Helm charts for 2 pilot apps
Week 7
Deploy Remaining 4 Apps Through Pipeline & Terraform App Module
Objective
Onboard all remaining 4 apps through the established DevSecOps pipeline from DEV to production. Write and apply the Terraform app module to complete full IaC coverage.
Key Activities
  • Review and validate Dockerfiles and Helm charts for remaining 4 apps
  • Push all 4 apps through CI/CD pipeline: DEV → quality gate → Team Lead approval → STG/PRD
  • Resolve any pipeline failures per app (SonarQube findings, security scan issues)
  • Write and apply Terraform app module for all 6 apps
  • Confirm all Terraform modules applied cleanly — no outstanding drift
Pre-requisites
  • CI/CD pipeline validated end-to-end for pilot apps (from W6)
  • Developer team has written Dockerfiles and Helm charts for remaining 4 apps
  • Pipeline approval workflow confirmed with Team Lead / SPV
Week 8
Database Migration to Production RDS
Objective
Deploy the production database cluster via Terraform and migrate all application databases using AWS DMS. Complete full-load migration, enable CDC replication, and cut over all apps to the new database endpoints.
Key Activities
  • Apply Terraform rds module — deploy production database cluster
  • Set up AWS DMS replication instance and source/target endpoints
  • Run full-load migration for all application databases
  • Enable DMS CDC replication — keep source and target in sync
  • Validate data integrity between source and target (row counts, checksums)
  • Cut over all 6 apps to new RDS endpoints — confirm all apps running correctly
  • Set Cloudflare DNS TTL to low value in preparation for W12 cutover
Pre-requisites
  • All 6 apps deployed to production through CI/CD pipeline (from W7)
  • Terraform rds module written and reviewed (from W5)
  • Source database credentials and schema documentation provided by Customer
  • DMS subnet group and security group pre-created
Week 9
Blue-Green Infra Setup, Validation & BAST 2 Sign-off
🏁 BAST 2 · 30%
Objective
Configure blue-green deployment infrastructure for all 6 apps. Run load tests and security reviews across the full stack. Formally close Month 2 with BAST 2 documentation and Customer sign-off.
Key Activities
  • Configure blue-green deployment infrastructure via CodeDeploy for all 6 apps
  • Validate blue-green traffic routing between Blue and Green environments
  • Run load tests on all 6 apps in production
  • Run security scan on all container images — resolve critical findings
  • Run Checkov static analysis on all Terraform IaC — resolve critical findings
  • Provide BAST 2 documentation and obtain Customer sign-off
Pre-requisites
  • All 6 apps connected to production RDS and running correctly (from W8)
  • All Terraform modules applied with no outstanding drift (from W7)
  • Customer stakeholder calendar blocked for BAST 2 presentation
🏁

BAST 2 — 30% Payment Milestone
All 6 apps on EKS production · DevSecOps CI/CD pipeline live · Database migration complete · Terraform IaC applied · Load test & security scan reports

Month 3 — Deliver  ·  Weeks 10–13

Week 10
Blue-Green Deployment Testing with Developer Team
Objective
Conduct blue-green rollout and rollback drills together with the developer team. Confirm the full deployment workflow is understood and can be executed independently before moving to observability setup.
Key Activities
  • Walk developer team through the blue-green deployment process end-to-end
  • Run rollout drills for all 6 apps together with the developer team
  • Run rollback drills — confirm team can execute rollback independently
  • Validate full pipeline approval flow: DEV → Team Lead approval → STG/PRD
  • Document and resolve any issues found during drill exercises
  • Confirm all 6 apps stable in production after drills
Pre-requisites
  • BAST 2 signed — Month 3 work can begin
  • Blue-green infrastructure configured for all 6 apps (from W9)
  • Developer team and Team Lead available for drill sessions this week
Week 11
Monitoring, APM & Observability Setup
Objective
Set up full observability for the production environment — CloudWatch dashboards, alarms, AWS X-Ray distributed tracing, and centralised log management. Only begins once production is confirmed stable from W10.
Key Activities
  • Enable CloudWatch Container Insights for the EKS cluster
  • Set up CloudWatch alarms covering EKS, RDS, ALB, and application metrics
  • Configure SNS notifications for alarm state changes
  • Build CloudWatch dashboards — infrastructure overview and per-app views
  • Deploy AWS X-Ray for distributed tracing across all 6 apps
  • Set up centralised log routing to CloudWatch Log Groups per app
  • Validate full observability stack — traces, metrics, and logs all flowing
Pre-requisites
  • All 6 apps confirmed stable after blue-green drills (from W10)
  • X-Ray integration approach confirmed with developer team
  • CloudWatch log group naming convention agreed with Customer
Week 12
AWS Backup & DNS / Firewall Migration from Cloudflare
Objective
Configure AWS Backup for data protection and validate restore. Migrate DNS from Cloudflare to Route 53 and replace Cloudflare firewall with AWS WAF and Shield Standard.
Key Activities
  • Configure AWS Backup plans for RDS and EBS with retention policies
  • Run a backup restore test and confirm data can be recovered
  • Import existing DNS zones into Route 53 and validate all records match
  • Update domain registrar NS records to point to Route 53
  • Monitor DNS propagation — confirm all apps resolve correctly via Route 53
  • Deploy AWS WAF WebACLs to ALB for all 6 apps
  • Enable AWS Shield Standard and decommission Cloudflare configuration
Pre-requisites
  • Monitoring and observability stack fully operational (from W11)
  • Cloudflare DNS TTL set to low value since W8
  • Domain registrar access credentials provided by Customer
  • WAF rule requirements agreed with Customer IT and security lead
Week 13
Documentation, Knowledge Transfer Workshop & Project Closure
🏁 BAST 3 · 40%
Objective
Deliver the complete documentation package and conduct a structured knowledge transfer workshop with the customer team. Verify operational independence. Obtain BAST 3 sign-off, revoke Xtremax access, and formally close the engagement.
Key Activities
  • Write and finalise all BAST 3 documents
  • Compile Operations Guide: runbooks, escalation matrix, backup and recovery procedures
  • Conduct knowledge transfer workshop — cover EKS operations, CI/CD pipeline, monitoring, DB, and DR procedures
  • Verify customer team can operate the environment independently
  • Final project presentation to Customer stakeholders
  • Provide BAST 3 documentation and obtain Customer sign-off
  • Remove all Xtremax access from Customer AWS accounts
  • Project closure — handover of all repositories, credentials, and documentation
Pre-requisites
  • DNS cutover complete and all apps stable on Route 53 (from W12)
  • AWS WAF and Shield Standard active (from W12)
  • Backup configuration and restore test completed (from W12)
  • All BAST 3 documents drafted and ready for review before the week starts
  • Customer team and stakeholders available for workshop and final presentation
🏁

BAST 3 — 40% Final Payment Milestone
