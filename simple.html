AWS Modernisation — Project Weekly Plan

PT Customer · Xtremax Engagement · 12-Week Delivery · 3 BAST Milestones

Month 1 · Foundation Month 2 · Build Month 3 · Deliver

Month 1 — Foundation

Week 1
Environment Discovery, Audit & RHEL Preparation
Objective
Fully understand the existing AWS environment. Produce a Discovery Report. Have all 6 RHEL target servers provisioned, hardened, and runtime-ready by Friday so migration can begin in Week 2.
Key Activities
  • Kick-off meeting — agree RACI, tools, access, communication cadence
  • Verify access to all 5 AWS accounts (Day 1 blocker)
  • Audit SCPs, OU structure, IAM users/roles, and MFA enforcement
  • Map VPC topology, Security Groups, Transit Gateway, and VPN routing
  • Interview all 6 app owners — capture dependencies, runtime versions, config
  • Provision RHEL 8.x EC2 instances in Dev/UAT for all 6 apps (Thursday)
  • Apply OS hardening on all 6 RHEL instances — CIS RHEL Level 1 baseline
  • Install runtime dependencies and verify SSM Agent on all RHEL hosts
  • Compile and present W1 Discovery Report to Customer IT lead (Friday)
Pre-requisites
  • PO issued — engagement clock starts Day 1
  • AWS console credentials for all 5 accounts provided before kick-off
  • 6 app owners introduced and available for 30-min interviews
  • Shadow engineers assigned and available from Day 1
  • Dev/UAT EC2 quotas and RHEL AMI access pre-confirmed
Week 2
Landing Zone Enforcement, Network Build & OS Migration Begins
Objective
Apply all security guardrails, fix compliance gaps, and build production network infrastructure. Deploy Apps 1 & 2 to RHEL with app owner sign-off by Friday.
Key Activities
  • Enforce missing SCPs: prevent root usage, enforce MFA, block S3 public access
  • Enable AWS Config, CIS conformance pack, and org-wide CloudTrail
  • Enable GuardDuty in all 5 accounts and Security Hub (FSBP standard)
  • Configure VPC Flow Logs and capture Security Hub baseline score
  • Create Production VPC: subnets (public/private/data), IGW, NAT GW ×2
  • Create VPC Interface Endpoints (S3, ECR, SSM, Secrets Manager)
  • Configure Security Groups for ALB, EKS, RDS tiers; tag all resources
  • Deploy App 1 to RHEL → functional smoke test → app owner sign-off
  • Deploy App 2 to RHEL → functional smoke test → app owner sign-off
Pre-requisites
  • W1 Discovery Report reviewed and approved by Customer IT lead
  • All 6 RHEL instances provisioned, hardened, and runtime-ready (from W1)
  • Remediation priority list agreed and signed off
  • Change window confirmed for network modifications (off-peak hours)
  • App owners for Apps 1 & 2 available Friday for sign-off sessions
Week 3
OS Migration Completion & EKS Prerequisites
Objective
Complete OS migration for Apps 3–6. Run load tests for all 6 apps. Write migration runbooks. Set up all EKS and IaC prerequisites so W4 cluster deployment is a clean, fast execution.
Key Activities
  • Deploy Apps 3 & 4 to RHEL → smoke tests → app owner sign-offs (Monday)
  • Deploy Apps 5 & 6 to RHEL → smoke tests → app owner sign-offs (Tuesday)
  • Run k6 load tests on all 6 apps — confirm p99 ≥ Windows baseline
  • Write migration runbook per app: deploy steps, env vars, rollback procedure
  • Run Security Hub FSBP — confirm ≥80% pass rate
  • Create EKS IAM service role and node instance profile in Production account
  • Create ECR repositories for all 6 apps with lifecycle policies
  • Create GitHub org + repos; set up Terraform remote state (S3 + DynamoDB)
  • Define Terraform module structure and scaffold /modules/vpc, /eks, /rds, /app
Pre-requisites
  • Apps 1 & 2 migrated to RHEL and signed off in W2
  • RHEL instances for Apps 3–6 provisioned and hardened from W1
  • App owners for Apps 3–6 available for validation sessions
  • GitHub organisation access granted by Customer
Week 4
BAST 1 Documentation, Sign-off & EKS Cluster Deployment
🏁 BAST 1 · 30%
Objective
Formally close Month 1 with Customer sign-off and invoice. Deploy the production EKS cluster (control plane, node groups, add-ons) and apply Terraform vpc + eks-cluster modules before Month 2 begins.
Key Activities
  • Write Landing Zone Governance Report and OS Migration Documentation
  • SA internal review — add evidence appendices (screenshots, reports)
  • Present BAST 1 documents to Customer stakeholders (Wednesday)
  • Obtain Customer signature → submit 30% invoice (Wednesday)
  • Deploy EKS cluster via Terraform: control plane (K8s 1.30), OIDC, 3 managed node groups in 2 AZs (Thursday)
  • Install EKS add-ons: CoreDNS, kube-proxy, VPC CNI, EBS CSI, AWS Load Balancer Controller
  • Write and apply Terraform vpc module over existing VPC resources
  • Month 1 retrospective + confirm Month 2 app owner availability (Friday)
Pre-requisites
  • All 6 apps validated on RHEL with app owner sign-offs collected
  • Security Hub FSBP score ≥80% confirmed
  • EKS IAM roles, ECR repos, GitHub repos, Terraform remote state all ready (from W3)
  • Customer stakeholder calendar blocked for Wednesday BAST 1 presentation
🏁

BAST 1 — 30% Payment Milestone
Landing Zone Governance Report · OS Migration Documentation (6 apps) · App owner sign-off evidence · Security Hub FSBP compliance report

Month 2 — Build

Week 5
Containerise Apps 1 & 2 & Complete Network IaC
Objective
Containerise Apps 1 & 2 as the pilot. Establish Dockerfile and Helm patterns for all remaining apps. Configure IRSA, CloudFront, Secrets Manager. Begin Terraform rds module.
Key Activities
  • Configure IRSA roles for all 6 apps (IAM role per app via OIDC)
  • Validate kubectl access for all team members
  • Write Dockerfile for App 1 (multi-stage Node.js build) → Trivy scan → push to ECR
  • Write Helm chart for App 1 (Deployment, Ingress, HPA, ServiceAccount) → deploy to EKS
  • Write Dockerfile + Helm chart for App 2 → deploy to EKS
  • Configure CloudFront in front of ALB for Apps 1 & 2 (HTTPS only)
  • Configure Secrets Manager for DB credentials and API keys for Apps 1 & 2
  • Configure HPA for Apps 1 & 2 → validate auto-scale under load test
  • App owner sign-off sessions for Apps 1 & 2 on EKS
  • Write Terraform rds module (Aurora PostgreSQL, subnet group, Secrets Manager)
Pre-requisites
  • BAST 1 signed — Month 2 work can begin
  • EKS cluster deployed, all add-ons in Ready state (from W4)
  • Terraform vpc + eks-cluster modules applied — state clean (from W4)
  • ECR repositories and GitHub repos ready (from W3)
  • Docker base images agreed: node:20-alpine, php:8-fpm-alpine
Week 6
Containerise Apps 3–6 & Build CI/CD Pipeline
Objective
Containerise remaining 4 apps using the patterns established in W5. Build the full CI/CD pipeline (CodePipeline → CodeBuild → ECR → CodeDeploy). Apply Terraform rds module.
Key Activities
  • Containerise Apps 3, 4, 5 & 6 — Dockerfile + Helm charts → deploy to EKS
  • Configure HPA for all 4 apps and validate auto-scale
  • App owner sign-off sessions for Apps 3–6 on EKS
  • Build CodePipeline: Source (GitHub) → CodeBuild → ECR → CodeDeploy to EKS
  • Add SonarQube quality gate to CodeBuild — block pipeline on critical issues
  • Add ECR image scan to CodeBuild — block on HIGH/CRITICAL vulnerabilities
  • Apply Terraform rds module: Aurora PostgreSQL writer + reader, RDS instance
  • Test CI/CD pipeline end-to-end for all 6 apps
Pre-requisites
  • Apps 1 & 2 Dockerfile and Helm patterns validated and approved (from W5)
  • Terraform rds module written and reviewed (from W5)
  • CodePipeline/CodeBuild IAM roles pre-created
  • SonarQube server accessible from CodeBuild VPC
Week 7
Database Migration & Terraform IaC Completion
Objective
Migrate all application databases to Aurora PostgreSQL and RDS using AWS DMS. Complete all 4 Terraform modules. Confirm terraform plan = 0 changes with all secrets in Secrets Manager.
Key Activities
  • Set up AWS DMS replication instance and source/target endpoints
  • Run DMS full-load migration for each application database
  • Enable DMS CDC (change data capture) for near-zero-downtime cutover
  • Validate data integrity post-migration (row counts, checksums)
  • Update all 6 app connection strings → point to new Aurora/RDS endpoints
  • Move all DB credentials to Secrets Manager — remove hardcoded secrets from code
  • Write and apply Terraform app module for all 6 apps
  • Confirm all 4 modules applied: terraform plan = 0 changes
Pre-requisites
  • All 6 apps deployed and running on EKS (from W6)
  • Aurora PostgreSQL and RDS instances deployed via Terraform rds module (from W6)
  • Source DB credentials and schema documentation provided by Customer DBA
  • DMS subnet group and security group pre-created
Week 8
Load Testing, Security Review & BAST 2 Sign-off
🏁 BAST 2 · 30%
Objective
Validate full system performance under load. Complete security scan of containers and IaC. Obtain BAST 2 sign-off. Prepare Cloudflare DNS for Week 11 cutover.
Key Activities
  • Run full k6 load tests on all 6 apps — validate performance SLAs
  • Run ECR image scans on all 6 app containers — resolve HIGH/CRITICAL findings
  • Run Checkov static analysis on all Terraform IaC — resolve critical findings
  • SA final review of EKS, network, and database architecture
  • Compile BAST 2 evidence package (test reports, security scan results, SA sign-off)
  • Present BAST 2 to Customer stakeholders → obtain signature → submit 30% invoice
  • Set Cloudflare DNS TTL → 60 seconds in preparation for W11 cutover
Pre-requisites
  • All 6 apps running on EKS with CI/CD pipeline operational
  • All databases migrated and apps pointing to Aurora/RDS
  • terraform plan = 0 changes confirmed across all 4 modules
  • Customer stakeholder calendar blocked for BAST 2 presentation
🏁

BAST 2 — 30% Payment Milestone
All 6 apps on EKS · CI/CD pipeline live · Database migration complete · IaC (4 Terraform modules) applied · Load test & security scan reports

Month 3 — Deliver

Week 9
Blue-Green Deployment & APM / Monitoring Setup
Objective
Configure CodeDeploy blue-green for all 6 apps and verify rollback completes in under 5 minutes. Set up X-Ray distributed tracing, CloudWatch Container Insights, and Fluent Bit log routing.
Key Activities
  • Configure CodeDeploy blue-green for all 6 apps (10% → 50% → 100% traffic shift)
  • Run rollout drill for all 6 apps and verify deployment completes cleanly
  • Run rollback drill for all 6 apps — confirm rollback completes in <5 minutes
  • Deploy X-Ray DaemonSet to EKS — instrument all 6 apps for distributed tracing
  • Enable CloudWatch Container Insights for EKS cluster
  • Deploy Fluent Bit DaemonSet — route logs to CloudWatch Log Groups per app
  • Validate full observability stack: traces, metrics, and logs flowing in CloudWatch
Pre-requisites
  • BAST 2 signed — Month 3 work can begin
  • CI/CD pipeline operational for all 6 apps (from W6)
  • All apps stable on EKS in production with load tests passed
  • X-Ray SDK integrated into app code (coordinate with dev team)
Week 10
CloudWatch Dashboards, Alarms & Backup Configuration
Objective
Build operational dashboards and alarms for all 6 apps and infrastructure. Configure AWS Backup with tested restore. Import Route 53 hosted zones in preparation for W11 DNS cutover.
Key Activities
  • Create ≥20 CloudWatch alarms covering EKS, RDS, ALB, and app-level metrics
  • Configure SNS → notify Teams and email on alarm state change
  • Build CloudWatch dashboards: per-app and infrastructure overview
  • Configure AWS Backup plans for EBS, RDS, and S3 with retention policies
  • Run RDS point-in-time restore test — confirm RTO ≤ 2 hours
  • Import existing DNS zones into Route 53 hosted zones
  • Validate Route 53 records match Cloudflare configuration
Pre-requisites
  • X-Ray, Container Insights, and Fluent Bit all operational (from W9)
  • Cloudflare DNS TTL set to 60s (from W8)
  • AWS Backup vault created with appropriate KMS key
  • Route 53 hosted zone creation approved by Customer IT lead
Week 11
DNS & Firewall Migration from Cloudflare to AWS
Objective
Cut DNS over from Cloudflare to Route 53. Deploy AWS WAF WebACLs and Shield Standard. Conduct final SA architecture review before BAST 3 documentation begins.
Key Activities
  • Update domain registrar NS records — point to Route 53 name servers
  • Monitor DNS propagation — validate all apps resolve correctly via Route 53
  • Deploy AWS WAF WebACLs to ALB and CloudFront for all 6 apps
  • Configure WAF managed rule groups (Core Rule Set, Known Bad Inputs, SQLi)
  • Enable AWS Shield Standard — confirm DDoS protection active
  • Decommission Cloudflare DNS and firewall configuration
  • SA final architecture review — validate end-to-end security and network posture
Pre-requisites
  • Route 53 hosted zones imported and validated (from W10)
  • Cloudflare DNS TTL at 60s for >72 hours (set in W8)
  • Domain registrar access credentials provided by Customer
  • WAF rule groups agreed with Customer IT and security lead
  • Rollback plan documented: re-point NS to Cloudflare if needed
Week 12
Final Documentation, Knowledge Handover & Project Closure
🏁 BAST 3 · 40%
Objective
Deliver the complete documentation package. Confirm shadow engineer independence through a live assessment. Obtain BAST 3 sign-off, revoke Xtremax access, and formally close the engagement.
Key Activities
  • Write and finalise all 5 BAST 3 documents (Architecture, CI/CD, DB, Monitoring, DNS/Security)
  • Compile Operations Guide: runbooks, escalation matrix, incident response, backup procedures
  • Shadow engineer independence test — perform a deployment and incident simulation unassisted
  • Final project presentation to Customer stakeholders and management
  • Address all final feedback and document sign-offs
  • Customer signs BAST 3 → submit final 40% invoice
  • Remove all Xtremax IAM users and access keys from Customer accounts
  • Project closure — handover of all credentials, repos, and documentation
Pre-requisites
  • DNS cutover complete and all apps stable on Route 53 (from W11)
  • WAF and Shield Standard active and configured (from W11)
  • All backups configured and restore test passed (from W10)
  • Shadow engineers assessed and confirmed operationally ready
  • Customer stakeholder calendar blocked for final presentation
🏁

BAST 3 — 40% Final Payment Milestone
